# Дизайн

## Структура приложения

Проект предоставляет небольшое FastAPI-приложение в `app/main.py`.
Вспомогательные модули разделяют ответственность:

- `app/models.py` определяет схемы запроса и ответа.
- `app/database.py` инициализирует SQLite и сохраняет заявки.
- `app/deduplication.py` нормализует имена и российские номера телефонов.
- `app/emailer.py` читает настройки из `.env` и отправляет SMTP-уведомления.
- `app/logging_config.py` настраивает файловое логирование в UTF-8.

## Модель данных

SQLite-таблица `leads` хранит обязательные поля:

- `id INTEGER PRIMARY KEY AUTOINCREMENT`
- `created_at TEXT NOT NULL`
- `name TEXT NOT NULL`
- `contact TEXT NOT NULL`
- `source TEXT NOT NULL`
- `comment TEXT NOT NULL`

Для детерминированной дедупликации таблица также хранит:

- `normalized_name TEXT NOT NULL`
- `normalized_contact TEXT NOT NULL`

Уникальный индекс на `(normalized_name, normalized_contact)` предотвращает
дубли строк для одного человека и одного нормализованного российского телефона.

## Дедупликация

`name` очищается от пробелов по краям и приводится к нижнему регистру. Телефон
нормализуется через извлечение цифр:

- `8XXXXXXXXXX` преобразуется в `+7XXXXXXXXXX`.
- `7XXXXXXXXXX` преобразуется в `+7XXXXXXXXXX`.
- `XXXXXXXXXX` преобразуется в `+7XXXXXXXXXX`.
- Уже нормализованный `+7XXXXXXXXXX` остается в этом формате.

Остальные форматы телефонов отклоняются с HTTP 400, потому что правило
дедупликации MVP зависит от российского телефона в формате `+7XXXXXXXXXX`.

Если приходит дубль, API возвращает id существующей заявки с `created: false` и
не отправляет повторное email-уведомление.

## Логирование

`events.log` записывается в UTF-8. В нем фиксируются:

- принятые новые заявки: `New lead saved: <id>.`
- дубли заявок
- успешная отправка email, пропуск отправки или ошибка отправки
- ошибки базы данных

## Email

SMTP-конфигурация читается из `.env` и переменных окружения процесса.
Переменные окружения имеют приоритет над значениями из `.env`.

Ожидаемые ключи:

- `SMTP_HOST`
- `SMTP_PORT`
- `SMTP_USERNAME`
- `SMTP_PASSWORD`
- `SMTP_FROM`
- `SMTP_TO`
- `SMTP_USE_TLS`

Если SMTP не настроен, сервис все равно сохраняет заявку и пишет понятную
строку в лог о том, что email был пропущен. Если SMTP настроен, но отправка
завершилась ошибкой, ответ по заявке остается успешным, а ошибка логируется.

## Обработка ошибок

- Невалидный JSON и ошибки валидации запроса возвращают HTTP 400.
- Отсутствующий или пустой `contact` возвращает HTTP 400.
- Неподдерживаемый формат телефона возвращает HTTP 400.
- Сбои SQLite возвращают HTTP 500 и логируются.

## Кодировка ответов

Все JSON-ответы должны явно указывать `charset=utf-8` в `Content-Type`, чтобы
кириллица корректно отображалась в Windows PowerShell и других клиентах,
которые не угадывают UTF-8 для `application/json`.
