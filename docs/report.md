# Отчет о практике

## 1. Кому и какую задачу решает проект

Проект предназначен для небольшого бизнеса, который получает заявки из форм,
соцсетей или таблиц и хочет не терять их по дороге. Сервис автоматически
принимает новую заявку через `POST /lead`, сохраняет ее в базе и оставляет
понятный след в логах и уведомлениях.

## 2. Входные данные

В `POST /lead` приходит JSON с полями:

- `name` - имя клиента
- `contact` - контакт для связи
- `source` - источник заявки
- `comment` - комментарий клиента

Обязательное поле - `contact`. Для успешной обработки оно должно содержать
непустое значение. В проекте также используется нормализация имени и телефона
для простой дедупликации.

## 3. Выход / результат

Для бизнеса результатом становится новая запись в SQLite-базе и событие в
`events.log`. Если SMTP-настройки заполнены, сервис дополнительно отправляет
email-уведомление менеджеру о новой заявке.

## 4. Архитектура

Поток работы устроен так:

`endpoint` -> `validation` -> `db` -> `notify / log`

- `endpoint` принимает `POST /lead`.
- `validation` проверяет JSON, обязательные поля и формат телефона.
- `db` сохраняет заявку в SQLite и не дает создавать дубли.
- `notify` отправляет email через `smtplib`, если SMTP настроен.
- `log` фиксирует сохранение заявки, дубль, ошибки и статус уведомления.

## 7. Ошибки и как решал

- PowerShell по умолчанию ломал `curl`-вызовы, поэтому для документации
  добавил `curl.exe` и UTF-8-safe вариант с временным JSON-файлом.
- В Windows PowerShell кириллица могла превращаться в `?????`, поэтому в
  документации описал передачу JSON как UTF-8 bytes.
- Отдельно проверял содержимое SQLite через `hex(...)`, чтобы отличать проблему
  хранения данных от проблемы отображения в DBeaver.
- Для 400-ошибок сделал понятные JSON-ответы, чтобы пользователь сразу видел,
  что не так с запросом.
- Для 500-ошибки добавил запись в лог, чтобы сбой базы не терялся без следа.
