# Технический обзор

## Архитектура

MVP состоит из небольшого FastAPI-приложения и нескольких модулей:

- `app/main.py` - HTTP endpoint, обработка ошибок, вызов хранения и уведомлений.
- `app/models.py` - Pydantic-схемы входа и выхода.
- `app/deduplication.py` - нормализация имени и телефона.
- `app/database.py` - SQLite-схема, вставка заявки и поиск дублей.
- `app/emailer.py` - SMTP-уведомление через `smtplib`.
- `app/config.py` - чтение `.env` и переменных окружения.
- `app/logging_config.py` - запись событий в UTF-8 файл.

## Поток `POST /lead`

1. FastAPI валидирует JSON по `LeadRequest`.
2. Имя нормализуется через `strip().lower()`, телефон приводится к
   `+7XXXXXXXXXX`.
3. SQLite инициализируется при запросе, чтобы локальный запуск не требовал
   отдельной миграции.
4. Заявка вставляется в `leads`. Уникальный индекс на
   `(normalized_name, normalized_contact)` защищает от дублей.
5. Новая заявка пишет `New lead saved: <id>.` в `events.log`.
6. Если SMTP настроен, отправляется email. Результат отправки всегда пишется
   в лог.

## Ручные HTTP-запросы

Для ручной проверки используется фактический endpoint проекта: `POST /lead`.
Примеры команд `curl -s ... | jq` находятся в `docs/manual_requests.md`.

Команда вида `POST /chat/message` не применяется в этом MVP: такого endpoint
нет в активной OpenSpec-спеке.

## Ограничения

- Один бизнес-сценарий: новая заявка.
- Одна SQLite-база.
- Без фоновой очереди: email отправляется синхронно после сохранения заявки.
- Если email не отправился, сохраненная заявка не откатывается.
- Дедупликация рассчитана на российские телефоны.
